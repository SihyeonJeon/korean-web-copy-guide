# KOREAN_WEB_COPY.md

AI 코딩 에이전트용 한국어 웹 문구 규칙.

한국어 제품 페이지, SaaS 페이지, 대시보드, 포트폴리오 페이지, UI 마이크로카피에 사용한다. 이 파일은 입력 맥락 분류, 공개 문구 변환, 화면 위치별 문장형, 줄바꿈, 단어 선택, 점검 절차를 다룬다. 시각 디자인, 제품 사실, 주장, 레이아웃은 다루지 않는다.

## 목적

AI가 만든 한국어 웹 문구는 페이지 문구 대신 프롬프트를 드러내는 경우가 많다.

자주 발생하는 실패:

- 요청 충족 선언: `깔끔하게 만들었습니다`, `요청하신 사이트입니다`, `이것은 ... 하는 페이지입니다`
- 작업 지시 유출: `공개 버전에서는 ... 제외`, `로컬 파일은 올리지 않습니다`
- 참고 맥락의 제목화: 사용자가 말한 `A를 위한`을 결과물 이름이나 공개 설명으로 승격
- 어색한 무주어 평서문: `제공합니다`, `지원합니다`, `관리합니다`, `않습니다`
- 남의 제품 해설문: `이 제품은 ...`, `이 서비스는 ...`
- 다짐처럼 읽히는 선언형: `...하지 않습니다`, `...을 줄입니다`, `...을 해결합니다`
- 무심한 줄바꿈: 한 음절 또는 짧은 조사만 다음 줄에 남는 상태
- 당연한 설명: `사용자는 버튼을 눌러 이동할 수 있습니다`
- 사적 영어 이름: 직접 만든 용어를 표준 제품 용어처럼 사용
- 흔들리는 톤: 캐주얼, 기업 홍보, 튜토리얼, 자기 PR이 한 화면에 섞인 상태

목표:

- 입력 맥락과 공개 문구의 분리
- 자기 설명보다 제품 사실
- 구현 설명보다 사용자 작업
- 자체 영어 이름보다 보편적인 한국어 제품 용어
- 의미 단위 줄바꿈
- 짧고 훑어보기 쉬운 문구
- 화면 위치별 문장 형태 구분
- 제품 해설문보다 작업 라벨과 상태 라벨

## 관찰 패턴

국내 서비스 웹 문구는 대체로 페이지를 설명하지 않고 사용자의 다음 작업을 앞에 둔다. 소비자 서비스는 `조회`, `확인`, `받기`, `찾기`처럼 일상 동사를 먼저 쓰고, B2B 서비스는 `관리`, `추적`, `연동`, `검증`, `운영`처럼 작업 단위를 먼저 쓴다.

대기업/엔터프라이즈 페이지는 문장이 길어지더라도 기능 범위, 구축 범위, 운영 범위를 명시한다. 단순히 `AI로 혁신`이라고 말하기보다 어떤 데이터, 문서, 모델, 프로세스, 채널을 다루는지 적는다.

SaaS 페이지는 문제를 말할 때도 감정 표현보다 손실, 위험, 규정, 병목, 반복 작업을 먼저 둔다. 증거는 숫자, 인증, 고객 유형, 지원 연동, 배포 환경처럼 검증 가능한 단위로 제시한다.

공통적으로 피해야 할 것은 페이지 제작 과정을 드러내는 문장이다. 공개 웹 문구에서는 `만들었습니다`, `반영했습니다`, `구성했습니다`보다 사용자가 보는 화면, 상태, 결과, 제약이 먼저 온다.

줄바꿈은 문장 길이보다 의미 단위가 우선이다. 제목은 짧은 명사구로 줄이고, 설명은 화면 폭에 맞춰 자연스럽게 흐르게 둔다. 한 글자, 조사, 숫자 단위, 약어가 혼자 남는 줄은 문구 또는 레이아웃을 조정해 제거한다.

내부 제작 맥락은 공개 문구로 올리지 않는다. 서비스 페이지는 `공개용으로 정리했습니다`, `내부 자료는 제외했습니다`처럼 제작 지시를 설명하지 않는다. 대신 사용자가 볼 수 있는 제품명, 기능명, 작업 경로, 지표, 상태, 제한 조건만 남긴다.

## 문맥 파일 역할

| 파일 | 담당 |
| --- | --- |
| 제품 brief | 제품 사실, 대상, 작업 흐름, 주장, 증거, 한계 |
| `DESIGN.md` | 색, 레이아웃, 글꼴, 간격, 모션, 컴포넌트 동작 |
| `SITE.md` | 섹션 순서, 페이지 구조, 내비게이션 모델 |
| `VOICE.md` | 특정 참조 문체가 있을 때 문장 리듬과 CTA 형태 |
| `KOREAN_WEB_COPY.md` | 입력 분류, 공개 문구 변환, 한국어 문장형, 줄바꿈, 금지 패턴 |

파일이 충돌하면 가장 좁은 역할의 파일을 따른다. 이 파일로 제품 사실을 새로 만들지 않는다.

## 사용 순서

1. 입력 분류: 프롬프트에 섞인 공개 사실, 참고 맥락, 작업 지시, 내부 제약을 나눈다.
2. 공개 문구 변환: 공개 가능한 사실만 화면 라벨, 제목, 본문, 상태값으로 바꾼다.
3. 화면 위치별 문장형: 첫 화면, 섹션, 카드, 버튼, 상태, 도움말에 맞는 길이와 형태를 고른다.
4. 점검: 금지 표현, 주어, 종결형, 줄바꿈, 영어 용어, 근거 없는 주장, 프롬프트 유출을 검색한다.

## 1. 입력 분류

프롬프트에는 공개 문구에 들어갈 사실과 에이전트가 처리해야 할 지시가 섞여 있다. 문구 작성 전에 입력을 먼저 분류한다.

### 1.1 입력 종류

| 입력 종류 | 예 | 처리 |
| --- | --- | --- |
| 공개 사실 | 제품명, 기능명, 대상 사용자, 도메인, 지표, 지원 범위 | 공개 문구에 사용 가능 |
| 화면 구조 | 첫 화면, 카드, CTA, 목록, 표, 내비게이션 | 레이아웃에 반영하고 문장으로 설명하지 않음 |
| 작업 지시 | 공개용으로 정리, GitHub 업로드, 로컬 파일 제외, 깔끔하게 개발 | 공개 문구에 사용 금지 |
| 참고 맥락 | Toss처럼, 대기업 웹페이지 말투, A를 위한, 포트폴리오용 | 공개 사실인지 내부 참조인지 판정 |
| 내부 제약 | 민감 정보 제외, 테스트 fixture, 배포 제외, 관리자 전용 | 사용자에게 필요한 정책일 때만 공개 |
| 어조 힌트 | 정제된, 차분한, 갤러리 같은, SaaS 같은 | 문구 길이와 단어 선택에만 반영 |

### 1.2 공개 문구 승격 기준

어떤 단어를 공개 제목이나 본문에 쓰기 전에 아래 순서로 판정한다.

1. 제품명, 서비스명, repo명인가
2. 사용자가 보는 대상 사용자, 도메인, 문제인가
3. 기능명, 데이터명, 작업명, 지표명인가
4. 화면에서 실제로 보이는 상태, 입력, 출력, 제한 조건인가
5. 아니면 작업자가 이해해야 할 내부 맥락인가

1-4이면 공개 문구로 쓸 수 있다. 5이면 공개 문구에서 제거한다.

### 1.3 참조어 처리

`A를 위한 프로젝트`라는 말에서 `A`가 결과물의 이름, 대상 사용자, 도메인, 기능명으로 명시된 경우에만 공개 문구에 사용한다. 단순 참고용 맥락이면 산출물 문구에 쓰지 않는다.

공개 제목에 쓰는 경우:

- 사용자가 명시한 제품명
- 사용자가 명시한 서비스명
- 사용자가 명시한 대상 사용자
- 사용자가 명시한 도메인 문제
- 실제 repo명 또는 패키지명

공개 제목에서 제거하는 경우:

- `공개용`
- `제출용`
- `포트폴리오용`
- `깃허브 업로드용`
- `참고용`
- `임시`
- `테스트용`
- `내가 보기 위한`
- `나중에 기억하려고`

### 1.4 입력 분류 예시

| 입력 | 공개 문구 후보 | 이유 |
| --- | --- | --- |
| `로컬 파일을 공개용으로 바꿔서 GitHub에 올려라` | `설치`, `실행`, `검증 명령` | `공개용`, `GitHub에 올려라`는 작업 지시 |
| `고객 상담 SaaS 참고해서 문의 관리 페이지를 만들어라` | `문의 관리`, `상담 이력`, `응답 상태` | 참고 서비스명보다 작업 단위 우선 |
| `A를 위한 프로젝트를 만들어라` | A가 공개 대상이면 `A 작업 흐름`, 아니면 실제 기능명 | A의 역할 판정 필요 |
| `깔끔한 대시보드` | `지표 비교`, `필터`, `내보내기` | `깔끔한`은 시각 힌트, 문구 아님 |
| `민감한 내용은 제외` | `예제 데이터`, `fixture`, `개인 설정 파일 제외` | 보안/저장소 정책에서만 사용 |

## 2. 공개 문구 변환

공개 문구는 사용자의 요청을 설명하지 않는다. 사용자가 보는 대상, 작업, 상태, 결과, 제한 조건으로 변환한다.

### 2.1 변환 순서

```text
프롬프트 맥락 -> 공개 사실 -> 화면 단위 -> 짧은 한국어 라벨
```

변환 절차:

1. 프롬프트에서 작업 지시와 내부 맥락을 제거한다.
2. 남은 공개 사실을 제품명, 기능명, 대상, 입력, 출력, 지표, 제한 조건으로 나눈다.
3. 화면 위치를 고른다.
4. 완전한 문장보다 명사구, 동작구, 상태 라벨로 줄인다.
5. 모바일 줄바꿈과 중복 설명을 확인한다.

### 2.2 요청 충족 선언 제거

생성된 페이지가 사용자 요청을 충족한다고 말하지 않는다.

금지:

- `깔끔하게 구성한 웹사이트입니다`
- `요청하신 내용을 반영했습니다`
- `이 페이지는 고객 관리를 쉽게 하는 서비스입니다`
- `AI를 활용해 문제를 해결하는 사이트입니다`
- `사용자 친화적으로 만든 대시보드입니다`

대체:

- 화면 제목
- 작업 명사
- 기능 명사
- 보이는 결과
- 지원하는 작업 흐름
- 측정 가능한 제약

| 회피 | 사용 |
| --- | --- |
| `깔끔하게 만든 고객관리 사이트입니다` | `고객 문의, 상담 이력, 응답 상태` |
| `AI 상담을 자동화하는 서비스입니다` | `반복 문의 자동 응답` |
| `데이터를 쉽게 확인할 수 있습니다` | `일별 전환, 채널, 캠페인 비교` |
| `사용자가 버튼을 눌러 이동할 수 있습니다` | `데모 보기` |
| `효율적인 업무를 지원합니다` | `승인 대기, 담당자, 처리 기한 표시` |

### 2.3 작업 지시 유출 방지

산출물은 사용자의 작업 지시를 설명하지 않는다.

금지:

- `공개 버전에서는 로컬 파일을 올리지 않습니다`
- `민감한 내용은 제외했습니다`
- `GitHub 업로드용으로 정리했습니다`
- `요청에 따라 README를 작성했습니다`
- `내부 자료를 공개용으로 바꾼 프로젝트입니다`
- `포트폴리오 제출을 위해 만든 자료입니다`
- `A를 참고해 만든 페이지입니다`

대체:

- 포함 파일 목록
- 실행 방법
- 입력/출력
- 공개 가능한 범위
- 라이선스
- 알려진 한계

| 회피 | 사용 |
| --- | --- |
| `공개 버전에서는 로컬 파일을 올리지 않습니다` | `포함: README, 예제 데이터, 실행 스크립트` |
| `민감한 내용은 제외했습니다` | `예제 데이터는 익명화된 fixture 사용` |
| `GitHub 업로드용으로 정리했습니다` | `설치, 실행, 검증 명령` |
| `포트폴리오 공개를 위해 작성했습니다` | `문제, 입력, 처리 단계, 결과, 한계` |

예외:

- 보안 문서에서 제외 정책을 명시해야 할 때
- 라이선스 또는 개인정보 처리 범위를 적어야 할 때
- 릴리즈 노트에서 변경 범위를 기록할 때

예외 상황에서도 작업자의 지시가 아니라 사용자에게 필요한 정책으로 쓴다.

| 작업자에게 하는 말 | 공개 문서용 |
| --- | --- |
| `로컬 파일은 올리지 않습니다` | `개인 설정 파일은 저장소에 포함하지 않음` |
| `공개용으로 바꿨습니다` | `예제 데이터와 재현 명령만 포함` |
| `A 참고용입니다` | `참고 자료: A` 또는 제거 |

### 2.4 주어와 관점

웹 문구에서 `이 제품은`, `이 서비스는`, `이 플랫폼은`을 주어로 두면 남의 제품을 해설하는 문장처럼 읽힌다. `우리는`을 주어로 두면 자기소개 또는 영업 문장처럼 읽힌다. 둘 다 기본값으로 쓰지 않는다.

금지에 가까운 주어:

- `이 제품은`
- `이 서비스는`
- `이 사이트는`
- `이 페이지는`
- `우리는`
- `본 솔루션은`
- `사용자는`

대신 작업 대상이 문장의 앞에 온다.

| 회피 | 사용 |
| --- | --- |
| `이 제품은 고객 문의를 자동으로 분류합니다` | `고객 문의 자동 분류` |
| `이 서비스는 상담 이력을 관리합니다` | `상담 이력 관리` |
| `우리는 더 안전한 AI 응답을 제공합니다` | `정책 기준을 벗어난 답변은 발송 전 차단` |
| `사용자는 리포트를 확인할 수 있습니다` | `리포트 보기` |
| `이 플랫폼은 데이터를 실시간으로 보여줍니다` | `실시간 지표 표시` |

완전한 문장이 필요하면 주어를 제품이 아니라 도메인 객체로 둔다.

| 어색한 문장 | 더 나은 문장 |
| --- | --- |
| `이 제품은 승인 대기 건을 보여줍니다` | `승인 대기 건이 담당자와 기한별로 표시됩니다` |
| `이 서비스는 답변을 검증합니다` | `답변은 정책 문서와 금지어 기준으로 검증됩니다` |
| `이 대시보드는 전환율을 비교합니다` | `전환율은 캠페인, 채널, 코호트 기준으로 비교됩니다` |

다만 위 문장도 카드나 첫 화면에서는 길다. 가능하면 다시 라벨로 줄인다.

- `승인 대기 건 표시`
- `정책 문서 기반 답변 검증`
- `캠페인별 전환율 비교`

### 2.5 종결형과 부정문

`~합니다`, `~습니다`, `~는다`, `~않는다`, `~않습니다`는 한국어 웹 화면에서 쉽게 보고서, 매뉴얼, 자기 다짐처럼 읽힌다. 특히 주어가 없으면 더 어색하다.

화면 문구에서 피할 종결:

- `제공합니다`
- `지원합니다`
- `관리합니다`
- `분석합니다`
- `해결합니다`
- `줄입니다`
- `가능합니다`
- `하지 않습니다`
- `않습니다`
- `않는다`

대체 방식:

| 종결형 | 대체 |
| --- | --- |
| `문의를 관리합니다` | `문의 관리` |
| `반복 작업을 줄입니다` | `반복 작업 자동화` |
| `오류를 방지합니다` | `오류 조건 사전 확인` |
| `허용하지 않습니다` | `허용되지 않는 값` |
| `변경할 수 있습니다` | `변경 가능` 또는 `권한 변경` |
| `데이터를 분석합니다` | `데이터 분석` 또는 `채널별 지표 비교` |

부정문은 `~않습니다`로 쓰면 금지 선언이나 다짐처럼 읽힌다. UI에서는 상태나 조건으로 바꾼다.

| 회피 | 사용 |
| --- | --- |
| `정책에 맞지 않는 답변은 발송하지 않습니다` | `정책 기준 미통과 답변 발송 차단` |
| `권한이 없으면 접근할 수 없습니다` | `접근 권한 없음` |
| `필수 값이 없으면 저장되지 않습니다` | `필수 값 누락` |
| `연동에 실패하면 데이터를 가져오지 않습니다` | `연동 실패 시 데이터 수집 중단` |
| `지원하지 않는 파일은 업로드되지 않습니다` | `지원하지 않는 파일 형식` |

부정의 원인과 다음 행동이 필요하면 두 줄로 나눈다.

```text
접근 권한 없음
관리자에게 권한 요청
```

```text
필수 값 누락
캠페인 이름과 시작일 입력
```

`~합니다`가 허용되는 위치:

- 약관, 정책, 개인정보, 법률 고지
- 도움말 문서
- 관리자 설정 안내
- API 문서
- 긴 블로그 본문
- 회사 소개의 공식 문장

허용되는 경우에도 화면 제목과 카드 제목에는 쓰지 않는다.

### 2.6 당연한 말 제거

어떤 웹페이지에도 붙일 수 있는 문장은 삭제한다.

삭제:

- `쉽고 빠르게 사용할 수 있습니다`
- `업무 효율을 높일 수 있습니다`
- `원하는 정보를 확인할 수 있습니다`
- `사용자는 메뉴를 통해 이동할 수 있습니다`
- `다양한 기능을 제공합니다`
- `AI를 통해 더 나은 경험을 제공합니다`

실제 대상으로 대체:

| 일반 문구 | 구체 문구 |
| --- | --- |
| `업무 효율을 높입니다` | `승인 대기 건을 담당자와 기한별로 정렬` |
| `정보를 쉽게 확인` | `결제 실패, 재시도, 환불 상태 표시` |
| `다양한 기능` | `필터, 저장 보기, CSV 내보내기` |
| `AI로 자동화` | `정책 문서 기반 답변 초안 생성` |

### 2.7 단어 선택

보편적인 한국어 제품 용어를 우선한다.

| 영역 | 단어 |
| --- | --- |
| 내비게이션 | 홈, 제품, 기능, 요금, 사례, 문서, 문의 |
| 데이터 | 데이터, 지표, 리포트, 대시보드, 로그, 이력, 상태 |
| 작업 흐름 | 승인, 검토, 처리, 배정, 알림, 자동화, 연동 |
| 접근 권한 | 권한, 계정, 팀, 워크스페이스, 보안, 인증 |
| 분석 | 비교, 추적, 분석, 집계, 필터, 세그먼트 |
| AI | 모델, 답변, 추천, 요약, 검색, 검증, 규칙 |
| 비즈니스 | 고객, 상담, 결제, 주문, 캠페인, 매출, 전환 |

영어 사용:

- 이미 표준적으로 쓰이는 경우 허용: `API`, `SDK`, `SaaS`, `CRM`, `AI`, `MLOps`
- 독자가 섞여 있으면 첫 사용 때 한국어 설명 병기
- 사적인 영어 제품 범주 생성 금지
- 약한 아이디어를 기술적으로 보이게 하려고 영어 사용 금지

직접 만든 이름은 첫 사용 근처에서 정의하지 않는 한 사용하지 않는다.

새 용어가 필요하면 다음을 정의한다.

- 의미
- 입력
- 출력
- 해석 방법

예시:

```text
응답 차단률
검증 규칙을 통과하지 못해 발송 전 중단된 AI 응답의 비율
```

기본값:

- 작업 중심
- 방어적 설명 최소화
- 자기 홍보 최소화

권장:

- `확인`
- `관리`
- `비교`
- `설정`
- `연동`
- `추적`
- `분석`
- `자동화`
- `승인`
- `처리`
- `보관`
- `내보내기`
- `권한`
- `이력`
- `상태`

회피:

- `혁신적인`
- `압도적인`
- `완벽한`
- `최고의`
- `차원이 다른`

- `직관적인`: 실제 상호작용 설명 없이 사용

## 3. 화면 위치별 문장형

한국어 웹 문구는 모든 위치를 완전한 문장으로 쓰지 않는다. 많은 화면은 문장보다 라벨, 명사구, 동작구, 상태값으로 읽힌다.

### 3.1 위치별 기본 형태

| 위치 | 기본 형태 | 예시 |
| --- | --- | --- |
| 첫 화면 제목 | 핵심 작업 또는 결과 명사구 | `상담 이력과 자동 응답을 한 화면에서` |
| 첫 화면 보조 문구 | 대상과 기준을 담은 짧은 동작구 | `채널별 문의를 고객 기준으로 묶어 확인` |
| 섹션 제목 | 기능 또는 문제 명사구 | `반복 문의 자동 응답` |
| 카드 제목 | 4-14자 명사구 | `권한 설정`, `응답 검증` |
| 카드 본문 | 입력, 기준, 출력 | `정책 문서를 참조해 답변 초안 생성` |
| 버튼 | 다음 행동 | `리포트 보기`, `설정 열기` |
| 상태값 | 상태 라벨 | `승인 대기`, `처리 중`, `연동 실패` |
| 빈 상태 | 대상 없음 + 다음 행동 | `등록된 캠페인 없음`, `캠페인 추가` |
| 도움말/법률 | 완전한 문장 허용 | `관리자는 멤버 권한을 변경할 수 있습니다.` |

완전한 문장은 도움말, 정책, 약관, 튜토리얼, 상세 설명에서만 사용한다. 제품 소개 화면, 카드, 대시보드, 포트폴리오 표면에서는 라벨과 동작구를 우선한다.

### 3.2 문장 구조

한 문장에는 하나의 역할만 둔다.

화면 문구는 먼저 라벨로 가능한지 확인한다. 라벨로 충분하면 완전한 문장으로 늘리지 않는다.

권장 구조:

- `[대상]을 [동작]`
- `[데이터]를 [기준]별로 [동작]`
- `[상태]와 [이력]을 한 화면에서 [동작]`
- `[팀 A]와 [팀 B]가 같은 [자료]로 [동작]`
- `[조건]이 맞지 않으면 [제한 동작]`
- `[문제 상황]을 줄이기 위해 [검증 기준]을 적용`

예시:

- `계약 상태와 결제 이력을 한 화면에서 확인`
- `캠페인, 채널, 코호트별 전환 비교`
- `승인 전 변경 내역을 로그로 보관`
- `정책에 맞지 않는 응답은 발송 전 차단`

페이지를 바깥에서 설명하는 문장 구조는 피한다.

- `이 사이트는 ...`
- `이 서비스는 ...`
- `이 기능은 ... 가능하게 합니다`
- `사용자는 ... 할 수 있습니다`
- `우리는 ... 제공합니다`
- `요청하신 ...`

예외:

- 법률 고지
- 도움말 문서
- API 문서
- 제품 용어집
- 새 기술 용어의 첫 정의

예외 상황에서도 직접 정의를 우선한다.

- `워크스페이스: 팀, 권한, 결제 정보를 묶는 작업 단위`

### 3.3 제목

제목은 훑어보기용 표지처럼 작동해야 한다.

명사구 또는 짧은 동사구를 우선한다.

- `반복 문의 자동 응답`
- `캠페인별 성과 비교`
- `권한과 변경 이력`
- `결제 실패 알림`
- `문서 기반 답변 검증`

회피:

- `더 나은 경험을 위한 기능`
- `쉽고 편리한 서비스`
- `비즈니스를 혁신하는 솔루션`
- `고객을 위한 특별한 기능`
- `사용자 친화적인 인터페이스`

권장 길이:

- 첫 화면 제목: 시각적 한 줄 기준 한국어 8-22자
- 섹션 제목: 6-18자
- 카드 제목: 4-14자
- 버튼 문구: 2-8자

제목이 두 줄을 넘으면 제목과 본문으로 나눈다.

### 3.4 본문

본문은 제약, 작업 흐름, 증거, 다음 행동을 추가할 때만 둔다.

좋은 본문에 들어가는 정보:

- 표시되는 데이터
- 지원되는 동작
- 동작을 제한하는 조건
- 존재하는 연동
- 보이는 지표 또는 상태
- 줄어드는 위험

다음 내용만 말하는 본문은 삭제한다.

- 페이지가 쉽다는 말
- 디자인이 깔끔하다는 말
- 서비스가 유용하다는 말
- 사용자가 클릭할 수 있다는 말
- 제품이 업무를 돕는다는 말
- AI가 더 좋게 만든다는 말

패턴:

```text
제목: [작업/결과 명사]
본문: [입력], [기준], [출력/상태]
CTA: [행동 동사]
```

예시:

```text
반복 문의 자동 응답
상품 정보, 배송 정책, 교환 기준을 참조해 상담 초안을 생성
규칙 설정
```

### 3.5 버튼과 링크

버튼은 화면의 품질이 아니라 다음 행동을 말한다.

권장:

- `데모 보기`
- `요금 확인`
- `문의하기`
- `설정 열기`
- `리포트 보기`
- `파일 업로드`
- `초대하기`
- `내보내기`
- `다시 시도`

회피:

- `자세히 알아보세요`: 반복 사용
- `시작해보세요`: 시작 대상 없음
- `멋진 경험 시작하기`
- `지금 혁신하기`
- `깔끔한 화면 보기`

CTA 개수:

- 첫 화면: 기본 CTA 1개, 보조 CTA 1개까지
- 기능 카드: 0개 또는 1개
- 표 행: 상태나 경로가 바뀔 때만 링크 사용

### 3.6 섹션 작성법

첫 화면:

```text
[핵심 작업 또는 결과]
[대상, 위치, 조건 중 하나를 담은 한 문장]
[기본 행동] [보조 행동]
```

회피:

```text
[페이지 자체를 설명하는 문장]
[일반적인 장점]
[장식적인 CTA]
```

권장 형태:

```text
상담 이력과 자동 응답을 한 화면에서
채널, 이메일, 게시판 문의를 고객 기준으로 묶어 확인
데모 보기  요금 확인
```

기능 섹션:

```text
[기능 명사]
[입력]을 [기준]으로 [출력]
```

예시:

```text
규칙 기반 응답 검증
프로모션, 배송, 환불 기준을 벗어난 답변은 발송 전 차단
```

증거 섹션은 실제 증거만 사용한다.

허용되는 증거:

- 측정 지표
- 고객 유형
- 인증
- 배포 환경
- 지원 연동
- 벤치마크 조건
- 알려진 한계

근거 없는 증거 표현 회피:

- `많은 고객이 선택`
- `검증된 품질`
- `업계 최고`
- `완벽한 보안`

빈 상태:

```text
아직 등록된 [대상] 없음
[다음 행동]
```

오류 상태:

```text
[대상]을 불러오지 못함
네트워크 상태를 확인한 뒤 다시 시도
```

책임 전가, 농담, 모호한 사과는 피한다.

### 3.7 줄바꿈

한국어 웹 문구는 의미 단위 줄바꿈이 필요하다.

금지:

- 마지막 줄에 한 음절만 남는 상태
- 줄 첫머리에 조사만 남는 상태
- 하나의 명사구인 `고객 관리`를 `고객` / `관리`로 분리
- 영어 약어와 뒤따르는 명사의 분리
- 숫자와 단위의 분리

함께 둔다:

- `고객 문의`
- `상담 이력`
- `AI 상담`
- `전환율 12%`
- `API 키`
- `데이터 서버`
- `권한 설정`
- `결제 실패`

수동 줄바꿈 정책:

- `<br>`은 고정 첫 화면 문구나 캠페인 문구에만 사용
- 동적 제목, 카드 제목, 표 셀, CMS 본문에는 `<br>` 사용 금지
- 단어가 어색하게 걸리면 강제 줄바꿈보다 폭, 글자 크기, 문장 길이부터 조정

한국어 웹 페이지 CSS 기준:

```css
:root {
  word-break: keep-all;
  overflow-wrap: anywhere;
}

.hero-title,
.section-title,
.card-title {
  text-wrap: balance;
}

.body-copy,
.card-copy,
.table-cell {
  word-break: keep-all;
  overflow-wrap: break-word;
}
```

확인 해상도:

- 360px mobile
- 390px mobile
- 768px tablet
- 1280px desktop
- 1440px desktop

줄바꿈 점검:

- 한 글자짜리 마지막 줄 없음
- 조사만 남은 줄 없음
- 숫자와 단위 분리 없음
- 약어와 명사 분리 없음
- 360px에서도 첫 화면 문구 판독 가능
- 버튼은 의도한 경우가 아니면 줄바꿈 없음

### 3.8 위치별 톤 조정

한 화면에는 하나의 톤만 사용한다.

소비자 제품:

- 친근하지만 유치하지 않게
- `해보세요`, `확인하세요`, `받아보세요` 허용
- 기술 명사보다 일상 명사 우선

B2B SaaS:

- 차분하고 정확하게
- 지표, 작업 흐름, 역할, 권한 중심
- `관리`, `연동`, `추적`, `검증`, `내보내기`

엔터프라이즈/AI 플랫폼:

- 정확하고 운영 중심
- 프로세스, 모듈, 배포, 거버넌스 중심
- 제품 브랜드가 이미 소유한 경우가 아니면 장난스러운 은유 회피

포트폴리오 화면:

- 건조한 라벨과 증거
- 작성자 중심 설득 금지
- 프로젝트, 스택, 입력, 출력, 지표, 한계

### 4.3 에이전트 출력 계약

한국어 웹 문구 작성 시:

- 요청 충족 선언 없음
- 작업 지시를 공개 문구로 승격하지 않음
- `공개용`, `업로드용`, `제출용`, `참고용`은 결과물 이름이나 설명으로 사용하지 않음
- 포함/제외 기준은 사용자에게 필요한 정책 문서에서만 사용
- 법률/도움말 정의가 아니라면 `이 사이트는 ...` 없음
- 제품 해설문처럼 읽히는 `이 제품은`, `이 서비스는`, `본 솔루션은` 없음
- 카드, 제목, 버튼, 상태값에 `~합니다`, `~습니다`, `~는다`, `~않습니다` 없음
- 부정문은 상태 라벨 또는 조건 라벨로 변환
- 정의 없는 자체 영어 이름 없음
- 근거 없는 최상급 없음
- 대상 없는 일반 장점 없음
- 한 문장에 하나의 역할
- 제목은 명사구처럼 훑힘
- 버튼은 다음 행동을 말함
- 모바일 줄바꿈 확인
- 출처 페이지는 리듬과 구조 참고용, 재사용 문구 아님

최종 점검:

```text
1. 사용자 프롬프트가 드러난 문장이 있는가?
2. `공개용`, `업로드용`, `참고용` 같은 작업 지시가 제목이나 본문에 남아 있는가?
3. 사용자가 말한 `A를 위한`의 A가 공개 대상인지, 내부 참고인지 판정했는가?
4. 제품이나 작업 흐름 대신 페이지 자체를 설명하는 문장이 있는가?
5. `이 제품은`, `이 서비스는`, `본 솔루션은`처럼 남의 제품 설명으로 읽히는 주어가 있는가?
6. 제목, 카드, 버튼, 상태값에 `~합니다`, `~는다`, `~않습니다`가 있는가?
7. 부정문을 상태 라벨로 바꿀 수 있는가?
8. 일반 형용사를 구체 명사로 바꿀 수 있는가?
9. 모든 CTA가 실제 행동을 말하는가?
10. 모든 줄바꿈이 한국어 의미 단위를 지키는가?
11. 모든 영어 용어가 표준적이거나 정의되어 있는가?
12. 모든 주장이 제품 사실로 뒷받침되거나 삭제되었는가?
```
